Blocked → 2 states (your call: collapse to In queue + blocked)

All the “stalled / needs-attention / plain” nuances (verified in StatusIcon.tsx + server/.../issues.ts) fold into one red blocked. The only distinction kept is whether the block is self-healing.

ChipYou see it when…Self-heals?
In queueA blocking sub-task/dependency is currently running (or human-assigned / awaiting approval) — the parent is only waiting on its own live child.Yes — unblocks when the child finishes. No action needed.
blockedEverything else that’s genuinely stuck — a blocker in review with no one on it, unassigned/parked/cancelled, an agent down, or a manual block.No — needs a human (decision / triage / unblock).

In queue is the one worth keeping distinct — it means not actually stuck (a child is moving it forward). Everything else is just blocked (red). The richer covered/stalled/needs-attention signal still exists under the hood (blockerAttention) and could power tooltips/filters later, but the chip is binary.

Your question: active vs idle (verified in code)

active and idle are not meaningfully different — they should be collapsed. The agents table defaults to idle and is never set to active; active exists only as a dashboard display alias (it counts idle agents as “active”). So the resting state is one thing — labelled “idle” (gray) — and “running” (blue) when a heartbeat is executing. That idle + running mix is what a healthy org shows above.

Decisions — resolved ✓

Locked into the plan (✦ = updated this round):

✦ D1 — active collapses into one resting state, shown gray “idle” (reverted from green “online”).

✦ D2 — pending_approval is removed as an agent state. Hiring approval happens in the task thread (the hiring-plan task pauses as a purple in review task → approve there).

✦ D3 — terminated is removed from the agents list (hidden like archived companies; lives elsewhere / DB), rather than shown gray.

✓ D4 — issue/task chip + status-icon restyle → follow-up issue (paperclip-website project, its own workspace). PAP-75 ships agents only.

✦ D5 — agent error capsule re-added (red, blinking). Scenario: the agent is broken (e.g. not connected to a model) and failing every task — distinct from a task failing.

✦ D6 — blocked task states collapse to 2: In queue (blue, self-heals) + blocked (red).

✓ D7 — paused agent = amber (gray would collide with the gray idle state).

✦ D8 — All Projects page chips adopt the same brand chip system. 5 PROJECT_STATUSES map to: backlog (gray) · planned (amber) · in_progress (blue, liveness) · completed (green) · cancelled (gray, strikethrough). Only two colour changes from today: planned gray→amber (split from backlog) and in_progress yellow→blue. paused overlay = amber; archived stays hidden from the list. Awaiting board approval.

All decisions confirmed by the board. Final agents list = 4 states: idle (gray) · running (blue, pulse 1.6s) · paused (amber) · error (red, blinking). active→idle; pending_approval & terminated not listed.
